# Atmosphere-AI Production Deployment Guide

This document provides a step-by-step blueprint to deploy **Atmosphere-AI** to a production Linux system (such as a Raspberry Pi 4/5 running Raspberry Pi OS, or a generic Debian/Ubuntu server).

---

## 🔌 1. Hardware Connection & Wiring Guide

Atmosphere-AI supports reading local sensors directly via serial connection or remote nodes.

### Local Raspberry Pi GPIO Connection (PMS5003/SDS011)
To connect a PMS5003 particulate matter sensor directly to the Raspberry Pi GPIO pins, use the following wiring scheme:

| PMS5003 Pin | Pi GPIO Pin Name | Pi Physical Pin Number | Description |
| :--- | :--- | :--- | :--- |
| **Pin 1 (VCC)** | 5V Power | Pin 2 or 4 | 5V DC Power Input |
| **Pin 2 (GND)** | Ground | Pin 6, 9, 14, 20, or 25 | Common Ground reference |
| **Pin 4 (RX)** | UART TXD (GPIO 14) | Pin 8 | Serial Transmit from Pi to sensor |
| **Pin 5 (TX)** | UART RXD (GPIO 15) | Pin 10 | Serial Receive from sensor to Pi |

> [!WARNING]
> By default on Raspberry Pi, the primary serial port (`/dev/ttyS0` or `/dev/ttyAMA0`) is mapped to the serial console. You **MUST** disable the serial console in order to read the sensor:
> 1. Run `sudo raspi-config`.
> 2. Navigate to **3 Interface Options** -> **I6 Serial Port**.
> 3. Select **No** to *"Would you like a login shell to be accessible over serial?"*.
> 4. Select **Yes** to *"Would you like the serial port hardware to be enabled?"*.
> 5. Finish and reboot the Raspberry Pi.

## 🖥️ 2. Production Server Environment Setup

To secure your production deployment, run the application under a dedicated unprivileged user named `atmosphere`.

### A. Create the System User
```bash
# Create a system user without login privileges or home directory conflicts
sudo useradd -r -s /bin/false atmosphere

# Add the user to the dialout group to allow direct serial port reading
sudo usermod -aG dialout atmosphere
```

### B. Directory Structure & Permissions
Copy the application files into `/var/www/atmosphere-ai` and apply the proper ownership policies:
```bash
# Create application root directory
sudo mkdir -p /var/www/atmosphere-ai
sudo cp -r . /var/www/atmosphere-ai

# Create virtual environment inside application folder
cd /var/www/atmosphere-ai
sudo python3 -m venv venv

# Set ownership to the atmosphere system user
sudo chown -R atmosphere:atmosphere /var/www/atmosphere-ai
```

### C. Install Production Dependencies
Run pip as the `atmosphere` user inside the virtual environment:
```bash
sudo -u atmosphere /var/www/atmosphere-ai/venv/bin/pip install --upgrade pip
sudo -u atmosphere /var/www/atmosphere-ai/venv/bin/pip install -r /var/www/atmosphere-ai/requirements.txt
```

## ⚙️ 3. Gunicorn WSGI Server Setup

Do not run the built-in Flask development server (`app.run()`) in production. Instead, run the app using **Gunicorn**, a production-grade WSGI HTTP server.

> [!IMPORTANT]
> Because Atmosphere-AI runs an in-memory background worker thread for sensor polling, you **MUST** run Gunicorn with exactly **one worker process** (`-w 1`). Spawning multiple Gunicorn workers would create multiple background threads, leading to SQLite lock contentions and duplicate sensor read operations.

Run Gunicorn using the virtual environment interpreter:
```bash
# Test starting the application using Gunicorn manually
sudo -u atmosphere /var/www/atmosphere-ai/venv/bin/gunicorn --workers 1 --bind 127.0.0.1:5000 app:app
```

## ⚡ 4. Systemd Daemon Service Configuration

Configure systemd to manage the Gunicorn process, ensuring it starts automatically on system boot and restarts automatically if it crashes.

### A. Create the Service File
Create `/etc/systemd/system/atmosphere.service`:
```ini
[Unit]
Description=Atmosphere-AI Environment Dashboard
After=network.target

[Service]
User=atmosphere
Group=atmosphere
WorkingDirectory=/var/www/atmosphere-ai
ExecStart=/var/www/atmosphere-ai/venv/bin/gunicorn --workers 1 --bind 127.0.0.1:5000 app:app
Restart=always
RestartSec=5
# Load configurations from .env if present
EnvironmentFile=-/var/www/atmosphere-ai/.env

[Install]
WantedBy=multi-user.target
```

### B. Control the Service
```bash
# Reload systemd configuration
sudo systemctl daemon-reload

# Enable the service to start automatically on system boot
sudo systemctl enable atmosphere.service

# Start the service immediately
sudo systemctl start atmosphere.service

# Check the status of the service
sudo systemctl status atmosphere.service
```

### C. Inspection & Logs
To view the output and logs generated by Gunicorn or the background sensor threads in real-time, query the systemd journal:
```bash
# Stream the last 100 log lines with follow parameter
sudo journalctl -u atmosphere.service -n 100 -f
```

## 🌐 5. Nginx Reverse Proxy Setup (with Let's Encrypt SSL)

To expose the application securely over ports 80/443 with a custom domain name, configure Nginx as a reverse proxy.

### A. Install Nginx
```bash
sudo apt update
sudo apt install nginx -y
```

### B. Configure Nginx Server Block
Create a virtual host configuration file at `/etc/nginx/sites-available/atmosphere`:
```nginx
server {
    listen 80;
    server_name atmosphere.yourdomain.com; # Replace with your actual domain

    location / {
        proxy_pass http://127.0.0.1:5000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # Enable caching for static resources
    location /static/ {
        alias /var/www/atmosphere-ai/static/;
        expires 30d;
        add_header Cache-Control "public, no-transform";
    }
}
```

Enable the configuration and restart Nginx:
```bash
# Link the site config to sites-enabled
sudo ln -s /etc/nginx/sites-available/atmosphere /etc/nginx/sites-enabled/

# Remove default nginx site config if not needed
sudo rm /etc/nginx/sites-enabled/default

# Test Nginx syntax configuration
sudo nginx -t

# Reload Nginx server
sudo systemctl reload nginx
```

### C. Secure Nginx with Let's Encrypt HTTPS
Install Certbot and request a certificate:
```bash
sudo apt install certbot python3-certbot-nginx -y
sudo certbot --nginx -d atmosphere.yourdomain.com
```
Follow the interactive prompts to enable automatic HTTP-to-HTTPS redirect. Certbot will configure SSL parameters inside your Nginx configuration block automatically.

## 💾 6. Automated SQLite Database Backups

Atmosphere-AI logs environmental history into a local SQLite database file `atmosphere_ai.db`. Ensure regular backups are performed.

### A. Create a Backup Script
Create `/var/www/atmosphere-ai/backup_db.sh`:
```bash
#!/bin/bash
DB_FILE="/var/www/atmosphere-ai/atmosphere_ai.db"
BACKUP_DIR="/var/www/atmosphere-ai/backups"
TIMESTAMP=$(date +"%Y%m%d_%H%M%S")
BACKUP_FILE="$BACKUP_DIR/atmosphere_ai_$TIMESTAMP.sqlite3"

mkdir -p "$BACKUP_DIR"

# Perform an online safe copy of the SQLite database to prevent corruption during writes
sqlite3 "$DB_FILE" ".backup '$BACKUP_FILE'"

# Compress the backup file
gzip "$BACKUP_FILE"

# Delete backups older than 30 days to free up storage space
find "$BACKUP_DIR" -type f -name "*.gz" -mtime +30 -delete
```

Configure ownership and executable permissions:
```bash
sudo chown atmosphere:atmosphere /var/www/atmosphere-ai/backup_db.sh
sudo chmod +x /var/www/atmosphere-ai/backup_db.sh
```

### B. Configure Cron Schedule
Add a cron job under the `atmosphere` system user to execute the backup script daily at midnight:
```bash
# Open cron editor for the atmosphere user
sudo -u atmosphere crontab -e
```
Add the following line at the bottom of the file:
```cron
0 0 * * * /bin/bash /var/www/atmosphere-ai/backup_db.sh
```

## 🛠️ 7. Maintenance & Operations Troubleshooting

### SQLite Database Locks
If the background pipeline throws database lock exceptions (e.g. `sqlite3.OperationalError: database is locked`), check for duplicate processes running:
```bash
ps aux | grep gunicorn
# Ensure only one master process and one worker process is active.
```

### Resetting/Clearing Database History
If you want to clear historical readings and restart from scratch:
```bash
# Stop the daemon service
sudo systemctl stop atmosphere.service

# Remove database files
sudo rm /var/www/atmosphere-ai/atmosphere_ai.db
sudo rm /var/www/atmosphere-ai/atmosphere_model.joblib

# Start the daemon service (will auto-recreate and seed 24h of history)
sudo systemctl start atmosphere.service
```
